# ADR-0001: 楽天APIプロキシの応答をキャッシュしない

## ステータス

採用（PR-3 時点）

## 背景

正式販売化 実装指示書 PR-3 は、`functions/api/rakuten.ts` の公式API結果に
30〜120秒程度の短時間キャッシュを設けることを「検討」項目として挙げている。
キャッシュを入れない場合は理由を記録することになっている。

## 検討したこと

- Cloudflare Pages Functions は `caches.default`（標準 Cache API）を提供しており、
  実装自体は数行で可能。
- しかし `caches` は Cloudflare Workers ランタイム固有のグローバルで、
  本リポジトリの単体テスト（Vitest / Node環境）には存在しない。キャッシュ処理を
  無条件に組み込むと、価格取得という中核ロジックの単体テストが打てなくなるか、
  ランタイム分岐（Node向けフォールバック）を余分に持つ必要が生じる。
- 表示価格は「検索表示から推定」ではなく `公式API取得`（確定的な取得結果）として
  ラベルされる。短時間とはいえキャッシュを挟むと、実際の楽天側の価格・在庫更新と
  表示のあいだにズレが生じうる。物販・せどり用途では、この種のズレが「利益が出ると
  思って仕入れたら実際は値上がりしていた」という実害につながりやすい。
- 楽天商品検索APIの無料枠（1,000リクエスト/日、開発者登録のみ）は、
  本ツールが想定するスタンダードプラン運用（個人の物販リサーチ用途、1日あたり
  数十〜百件程度の検索）に対して十分な余裕がある。今の利用規模では、
  キャッシュによるリクエスト削減の効果よりも、価格鮮度を犠牲にするデメリットが上回る。
- 過剰なリクエストへの対策は、キャッシュではなく Cloudflare 側のレート制限機能
  （`docs/deployment-guide.md` の「Cloudflareでのレート制限設定」参照）で行う方針とした。

## 決定

`functions/api/rakuten.ts` は応答をキャッシュしない。すべてのレスポンスに
`cache-control: no-store` を設定する。

## 再検討の条件

以下のいずれかに該当する場合は、キャッシュ導入を再検討する。

- 楽天APIの無料枠（1,000リクエスト/日）に近い利用量が常態化した場合
- Cloudflare側のレート制限だけでは実運用上のコスト・安定性が確保できない場合
- Cloudflare Pages Functions 用の統合テスト環境（`caches` を含む）を別途整備できる場合

その場合も、TTLは短時間（30〜120秒程度）に限定し、`no_key` / エラー応答は
キャッシュ対象から除外する方針を維持する。
